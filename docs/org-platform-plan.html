§ 0Summary & recommendation

The short version: keep Flow Builder a static, browser-only app. Don't build a backend, a database, or a proprietary storage layer. Instead, make the folder the platform:

  • Maps become files in a shared folder — a SharePoint document library or OneDrive folder the organization already has, with one .flow.json file per map. Permissions, versioning, backup, retention and audit come free from M365.
  • Flow Builder grows a Workspace layer — the existing home gallery evolves from "flows in this browser's localStorage" into "maps in the connected folder": browse, facet, search, open, dashboard. The editor itself is untouched.
  • A sidecar index makes it fast — a hidden .flowbuilder/ folder inside the library holds a small machine-written catalog: one ~3 KB digest per map (metrics, people, metadata, links) plus a search index and precomputed aggregates. Dashboards and search read digests, never the full corpus, so 1,000+ maps stay instant.
  • Cross-map structure comes from the format, not a server — stable map IDs, a link node type, and "promote group → sub-map" turn today's groups into real process decomposition with metric roll-ups.
  • Dashboards are small JSON specs — a template (role-shaped layout + metrics) bound to a scope (a metadata query like area = "Claims"). They live in the same library, so sharing and permissions are just file sharing.

What I'm explicitly not recommending: a Node/Python service, a database, SPFx/SharePoint-native development, or real-time collaborative editing. Each is evaluated below; none earns its complexity at this stage. The one honest caveat: a browser-only architecture has a scaling ceiling (roughly 2,000–5,000 maps per workspace, single-org search only within libraries a user can read). The design keeps a clean seam so a thin indexing service can be added behind the same file contracts if you ever cross that ceiling — nothing built now gets thrown away.

§ 1Where the tool is today — and what stays intact

Everything below builds around the current tool. Nothing in the editor, the metric engine, the lenses, exports, or the Convert-docs import changes.

App
One self-contained index.html (~390 KB), static hosting on Vercel, vendor libs local. No backend, no build step.
Persistence
All flows autosave to localStorage (per browser, per device). Portable via JSON "Back up design" export / Import.
Model
{nodes, edges, groups, comments} + title/theme. CSV import marks fields measured; Speed / Capacity / Quality metrics, flow efficiency, and the bottleneck are derived client-side.
Structure
Collapsible Groups inside a map (collapse is render-only). No links between maps, no stable map identity, no shared library.
Sharing
One-way exports: PNG/JPG/PDF/standalone HTML viewer.

The three gaps that block organizational use are exactly the three things the plan adds: shared storage (localStorage is a dead end for orgs — per-device, invisible to colleagues, unmanaged), identity (a map has no permanent ID, so nothing can reference it), and a cross-map layer (search, links, roll-ups, dashboards).

Assumption challenged

The current localStorage library can't remain the source of truth once folders exist — two sources of truth guarantee divergence ("which copy is real?"). The plan keeps localStorage only as an offline cache / scratch space for unconnected users, with a one-time "move my flows into the workspace" migration.

§ 2Architecture options, compared honestly

Four realistic ways to give a browser app shared, organization-wide storage. They are not mutually exclusive — the recommendation is a sequence, not a single pick.

Option A — SharePoint / OneDrive via Microsoft Graph API

Flow Builder (still a static page) signs the user in with MSAL.js and talks to the Microsoft Graph REST API directly from the browser: list the library, read/write .flow.json files, use delta queries to learn what changed since last visit. Works from any browser, any device, honors SharePoint permissions natively.

Cost: an Entra ID app registration and admin consent — a real IT conversation. With Sites.Selected scoping, IT can grant access to specific sites only, which is the modern, security-friendly ask. Graph throttling (~excess of a few requests/sec sustained) matters only for the first-ever index build.

Option B — Local synced folder via the File System Access API

The user clicks "Connect folder" and picks their locally-synced OneDrive/SharePoint folder. The browser (Edge/Chrome) gets direct read/write access — no sign-in, no IT ticket, no consent, works offline, and the OneDrive sync client moves the bytes. This is the zero-friction on-ramp: a team can adopt it this quarter without asking anyone.

Cost: Chromium-only (fine for Edge-standard orgs; Safari/Firefox can't). Permission is re-confirmed per browser profile. Sync lag (seconds–minutes) means "shared" is eventually-consistent. No server-side anything — every client does its own reading.

Option C — Thin service (indexer + API)

A small hosted service watches the library (Graph webhooks), maintains the index and aggregates server-side, and serves search/dashboard queries. Solves cold-start, cross-library search, and scheduled aggregation properly.

Cost: hosting, deployment, secrets, monitoring, a second codebase, and — in most enterprises — a security review before anything can touch tenant data. It changes the product's nature from "a file you open" to "a system someone operates". Powerful, premature.

Option D — SharePoint-native (SPFx web part, SharePoint Search, Lists)

Rebuild the workspace as a SharePoint Framework web part; push metadata into library columns; lean on SharePoint's own search and Power BI. Deepest M365 integration, and dashboards could partly be Power BI.

Cost: SPFx toolchain and deployment model, tenant app catalog governance, a UI constrained by SharePoint's shell, and hard lock-in. SharePoint Search also can't see inside your JSON meaningfully (steps, people, tools), so you'd still need your own index. Rejected as the platform, though § 7 borrows one cheap idea from it (CSV aggregates Power BI can read).

The digest — the load-bearing idea

Opening 1,000 full maps to draw a dashboard is a non-starter (≈300 MB of JSON). So whenever a map is saved, Flow Builder also writes a digest: a ~2–4 KB extract with everything the cross-map layer needs and nothing it doesn't:

.flowbuilder/digests/<mapId>.json
{
  "mapId": "m_7f3a…", "title": "Claims intake", "path": "maps/claims-intake.flow.json",
  "rev": 41, "savedAt": "2026-07-11T14:02:11Z", "savedBy": "amartel",
  "meta":   { "area": "Claims", "team": "Intake", "owner": "J. Ortiz",
              "status": "active", "reviewedAt": "2026-06-02", "tags": ["customer-facing"] },
  "metrics":{ "steps": 34, "people": 12, "leadTimeH": 86, "touchTimeH": 9.5,
              "flowEff": 0.11, "capacityPerDay": 41, "rftPct": 78,
              "bottleneck": { "step": "Adjuster review", "why": "capacity" },
              "confidence": { "measured": 21, "modelled": 9 } },
  "links":  { "out": ["m_2c9d…"], "in": [] },
  "text":   { "steps": ["Receive claim", …], "people": ["Intake clerk", …],
              "tools": ["Guidewire", …], "notes": ["Rekeying between systems", …] }
}

Everything downstream — search, facets, dashboards, roll-ups, link resolution, governance — reads digests. The full map is only ever opened to edit it. This is the entire performance story, and it's why no server is needed.

Keeping the index fresh without a server

There is no daemon, so the rule is: writers update what they touched; readers repair the rest. Saving a map writes its digest in the same breath. Opening the Workspace runs a background reconcile — compare file mtimes/etags against the catalog, re-digest anything stale (a few files, typically), rebuild affected aggregates, and bump catalog.json with an optimistic-concurrency check (etag; on conflict, re-read and merge — digest writes are idempotent, derived purely from map content, so merges can't corrupt). Stale-by-minutes is the accepted trade for having no infrastructure, and for operational dashboards over process maps that staleness is invisible.

§ 4File format & metadata foundations

These format changes are the true phase 0 — cheap now, near-impossible to retrofit later. They're worth doing even if everything else in this document changes.

  1. Stable identity. Every map gets a permanent mapId (UUID) minted at creation and preserved forever — through rename, move, duplicate ("duplicate" mints a new one; "move" never does). Links, digests, dashboards and roll-ups all reference IDs, never paths, so reorganizing folders breaks nothing.
  2. schemaVersion + the migrate-on-load pattern the app already uses (migrateComments et al.), formalized.
  3. A meta block — the governance/organization spine: 
    "meta": {
  "owner": "J. Ortiz",            // a person, always — unowned maps get flagged
  "area": "Claims",               // org-defined hierarchy level(s)
  "team": "Intake",
  "status": "active",             // draft → active → under-review → retired
  "reviewedAt": "2026-06-02",
  "reviewCadenceDays": 180,
  "tags": ["customer-facing"],
  "custom": { "riskTier": "2" }   // fields defined by the org schema, below
}
  4. A link node type — a start/end/step node may carry linksTo: { mapId, nodeId? }. This single field is the whole § 6 hierarchy story's data model.
  5. Naming: <slug>.flow.json so files are recognizable in SharePoint, and a dedicated icon/preview later.

Custom metadata at scale: a schema the org owns

Free-form tags rot at 1,000 maps ("Claims" vs "claims" vs "CLM"). Fixed fields are too rigid across organizations. The middle path: a tiny .flowbuilder/schema.json per workspace where the org defines its own fields once — name, type (text / choice / person / date), allowed values, required-or-not — and the editor's map-details panel renders them as proper inputs with dropdowns. The Workspace then gets clean facets for free, and validation is gentle: unknown values are flagged in a governance view, never blocked at save time (blocking kills adoption).

Metadata model

How structured should map metadata be?

§ 5Search that sees inside the maps

SharePoint's own search treats a .flow.json as an opaque blob — it will never answer "which processes involve the Underwriting team and mention Guidewire?". Ours will, because the digest already extracts exactly the searchable substance of a map:

  • Fielded full-text over titles, step labels, people, tools/systems mentioned, note text (including waste/frustration/idea notes), group names, and metadata — via a MiniSearch-style in-browser index (fuzzy + prefix matching, ~30 KB library, no dependency risk).
  • Facets from metadata: area, team, owner, status, tags, custom fields — instant in-memory filtering over the catalog.
  • Metric queries as filters, not text: flow efficiency < 10%, bottleneck type = capacity, not reviewed in 12 months. These come straight off digest fields and compose with text search ("rekeying in Claims, active maps, FE < 15%").
  • Results that answer, not just point: each hit shows where it matched — the step card, the note, the person — so users rarely need to open the map at all.

Index shards live in .flowbuilder/search/ (split by area once large), cached in IndexedDB, patched incrementally when digests change. At the 1,000-map scale this is a 1–2 s first load and instant thereafter — comfortably inside "no noticeable lag". No decision needed here; this design follows from D2.

§ 6Hierarchy, links between maps, and roll-ups

This is the hardest design problem in the brief, and the answer starts from something you already built: the collapsible Group is a sub-process viewed from outside. A collapsed group renders as a single step; § 6 extends that idea across file boundaries.

Group vs. separate map — a rule of thumb users can hold

Keep it a group (same map)Make it its own map
Same owner and team as the parent flowDifferent owner, or its own review cadence
Only meaningful inside this flowReferenced by two or more parent flows (shared service, e.g. "Credit check")
Edited together with the parentImproved/measured independently
A handful of stepsBig enough to be a working session of its own

The tool enforces nothing here — it just makes moving between the two states effortless:

  • Promote group → map. One action on any group: its members move into a new .flow.json (with meta inherited, ownership prompted), and the group is replaced in the parent by a single linked step — a collapsed-group-style card carrying linksTo: {mapId}. Boundary edges become the child map's start/end.
  • Demote map → group is the exact inverse, for when a decomposition turns out to be overkill. Reversibility is what makes people willing to decompose at all.

Linked steps: one continuous flow, not disconnected diagrams

  • A linked step renders like today's collapsed group (striped border, lens-sized) but its chip shows the child map's live digest metrics — lead time, capacity, bottleneck — so the parent map is always honest without opening anything.
  • Double-click drills in; a breadcrumb (Claims intake ▸ Adjuster review) drills back out. End-of-map link nodes ("continues in: Payments") work the same for peer-to-peer chaining. Navigation resolves mapId → path through the catalog, so renames and moves never break a link.
  • Backlinks are computed, not authored. The digest records outbound links; the catalog inverts them. Every map shows "used by: 3 flows" for free, and a shared sub-process knows all its parents.

Metric roll-ups

Roll-ups reuse the existing metric engine's semantics, applied over digests (never full maps):

PillarRoll-up rule
SpeedLinked step's touch/lead time = child's totals; parent's lead time and flow efficiency then compute exactly as today.
CapacityLinked step's capacity = child's constraint capacity (its bottleneck). Min-propagation matches the unified bottleneck logic already in the app.
QualityRight-first-time multiplies through: parent RFT × child RFT.
PeopleSum FTE; count distinct names where named (digest keeps the roster).
ConfidenceMeasured/modelled counts carry up, so an exec dashboard can say how much of a number is real data — precedence stays measured > observed > modelled.

Above the map level, the same digests aggregate by metadata — team → area → org — which is all a § 7 dashboard is. Cycle rule: navigation links may form loops (real processes do); roll-up links must form a DAG. The promote action and the governance view both run cycle detection and flag (not block) offenders, excluding them from roll-ups until resolved.

Staying maintainable at hundreds of maps

  • Broken links can barely happen (ID-based, catalog-resolved); a deleted target shows as a flagged tombstone on the linked step, listed in governance.
  • Duplicates: near-duplicate titles and step-set similarity are cheap to compute over digests; surfaced as a governance hint ("these 2 maps look like the same process").
  • Orphans & ownership: maps with no owner, no links, or expired reviews surface in the same place. Curation becomes a visible, assignable activity rather than tribal knowledge.

§ 7Dashboards & role-based views

The trap the brief rightly fears is hard-coding an "exec view" and a "manager view". The way out is to make a dashboard a tiny piece of data, not a piece of software:

.flowbuilder/dashboards/claims-ops.json
{
  "template": "ops-health@1",                      // layout + widgets + detail level
  "scope":    { "area": "Claims", "status": "active" },   // a metadata query
  "pins":     ["m_7f3a…"],                         // hand-picked additions, optional
  "overrides":{ "widgets": { "quality": {"hidden": true} } },
  "sharing":  "file"                               // it's just a file — library ACLs apply
}
Template
Defines layout, widget set, metrics, filters and altitude for a kind of consumer. Versioned; org can add its own.
Scope
A metadata query resolved against the catalog at render time — new maps tagged area: Claims join the dashboard automatically, no re-configuration.
Overrides
A small diff on the template. Personal tweaks never fork the template itself.
Inheritance
Org standards → role template → shared team dashboard → personal overrides. Each layer is a file (or localStorage for purely personal ones); "reset to standard" = delete the diff.

Standard templates to ship (one per altitude, all from the same digests)

TemplateForAnswersSignature widgets
My processesIndividual contributor"What am I part of, and how is it doing?"Maps naming me; expectations per step; my flows' health chips
Ops healthManager"Where is today's constraint?"Bottleneck board, capacity vs demand, queue waits, waste/frustration notes feed
Compare teamsDirector"Which of my teams needs help?"Small-multiple team rows: speed / capacity / quality side by side, improvement activity
PortfolioVP / exec"Consistent measures across areas + is the estate governed?"Area scorecards (people, lead time, FE%, RFT%, top bottlenecks), coverage & review-status meters, data-confidence bar

Every widget reads digest/aggregate fields — so when the digest gains a field, every dashboard can use it. That's also the seam for the dashboard builder later: the builder is just a UI that composes widgets over the digest schema and writes one of these JSON files. It costs nothing now to keep that door open, and the standard templates prove the widget catalog first.

Cheap borrowed win

Because aggregates are plain files, emitting aggregates/*.csv alongside the JSON lets Excel and Power BI consume Flow Builder data with zero integration work — a quiet answer to "why not Power BI?" (answer: you can, on top of us) that execs tend to love.

§ 8Governance without a bureaucracy

Governance data is deliberately derived, not a parallel form to fill in — almost all of it falls out of the meta block and things users already do:

  • Ownership & freshness: owner, status, last edit, last review vs cadence → staleness flags ("review overdue by 40 days"), unowned-map list.
  • Coverage: which areas/teams have maps at all, and how much of the estate carries measured (vs modelled) data — the confidence counts the metric engine already tracks.
  • Improvement activity: the waste / frustration / idea notes are literally an improvement backlog. Counts, ages, and resolution over time become an activity feed and a widget ("12 wastes logged, 5 addressed this quarter") with zero new authoring.
  • Estate integrity (from § 6): broken/tombstoned links, cycles excluded from roll-up, probable duplicates, schema-invalid metadata — one curation queue, assignable by area.

All of it surfaces in two places: a governance panel in the Workspace, and governance widgets available to any § 7 template (the Portfolio template ships with them on). No decision needed — scope rides on D3/D5.

§ 9Multi-user reality: conflicts, not collaboration

Once maps are shared files, two people will eventually edit one at once. The honest question is how much machinery that deserves.

My read: Flow Builder maps are authored artifacts with one natural owner, edited in sessions, read by many — much closer to a document than to a Figma canvas. Real-time co-editing (CRDTs, presence cursors, a sync server) would be the single most expensive line item in this whole plan and would drag in the very backend we're avoiding. SharePoint's file versioning already provides recovery for the rare collision.

§ 10Live data linking — designed now, built later

The tool already has the right primitive: CSV import marks step fields as measured, with precedence measured > observed > modelled. The platform generalizes the source without touching that engine:

  • Phase now (cheap): a map's meta can reference a data file in the library (dataSource: "data/claims-weekly.csv"). Opening the map re-reads it — so any process that can drop a CSV into SharePoint (Power Automate from a queue system, a nightly export, a person) becomes a "live" feed with zero new code paths. Digest records data age → dashboards show freshness.
  • Phase later: a connector descriptor format (API pull, Graph Excel ranges, webhook- dropped files) behind the same measured interface. Whether connectors run in-browser or in the optional service is decided then — the map format doesn't change either way.

No decision needed now; the only commitment is keeping measured-source attribution in the digest, which § 3 already does.

§ 11Security & identity, plainly

  • The big win of files-in-M365: permissions, encryption at rest, DLP, retention, eDiscovery, audit and backup are all the tenant's existing machinery. Flow Builder never stores org data anywhere else — the static app remains data-free hosting.
  • Mode B (local folder) needs no consent at all: the app touches only a folder the user explicitly picked, under their existing rights. Easiest possible security story to tell.
  • Mode A (Graph) requires an Entra app registration. Ask for Sites.Selected (IT grants specific sites) rather than tenant-wide file scopes — slower first conversation, far easier approval, and it's the pattern security teams now expect.
  • The index inherits the library's ACLs because it lives inside the library (D2). One workspace = one permission boundary. The subtle leak to avoid: never build a cross-library index that could show User X a digest (metrics, people names!) from a library they can't read. Cross-boundary aggregation is exactly the point where the optional service — which can enforce per-user trimming — earns its existence.
  • People names in maps are personal data. Digests keeping rosters is what makes "find processes involving Maria" work — worth an explicit note in org onboarding docs, and a workspace setting to digest roles instead of names for stricter orgs.

§ 13Roadmap: build now vs. enable later

Phase 0 Format foundations

Small · unlocks everything · zero user-visible risk

  • mapId, schemaVersion, meta block, link-node field, digest writer + validate, migrate-on-load.

Exit: every saved map has identity, metadata and a digest; old flows migrate silently.

Phase 1 Workspace: connect · browse · search

The adoption phase — a team gets value with zero IT involvement

  • Storage adapter (per D1), home gallery → Workspace over the folder, catalog + reconcile loop, faceted browse, full search, localStorage migration.

Exit: a 1,000-map folder browses and searches with no perceptible lag.

Phase 2 Links, hierarchy & roll-ups

Per D4

  • Linked steps + drill-in/breadcrumb, promote/demote group ⇄ map, digest-driven roll-ups, backlinks, tombstones + cycle flags.

Exit: a parent map shows live child metrics; deleting/renaming/moving never breaks a link.

Phase 3 Standard dashboards + governance

Aggregates + the four templates of § 7, governance panel of § 8, CSV export of aggregates

Exit: a VP opens Portfolio and sees consistent, confidence-annotated measures across areas.

Phase 4 Dashboard builder & role templating

Widget-catalog composer over the digest schema; template inheritance & org standards; sharing polish

Phase 5 Live connectors · optional thin service

Connector descriptors (§ 10); service only if an org crosses the browser ceiling (§ 12) — same file contracts either way
